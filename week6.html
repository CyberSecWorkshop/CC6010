Operation Phantom Coin
Bitcoin Investigation

A suspect paid for stolen payment card data using Bitcoin. Your task is to follow the money — trace three transactions, read the wallet file from the seized laptop, correlate browser history with payment dates, and write a forensic report.

Tools: Kali Linux · Python 3
Case ID: CC6010-2025-047
Duration: 2 hours
🧠 WHAT ARE WE DOING? — READ FIRST

Bitcoin is often called anonymous. It is not. Every Bitcoin payment is permanently recorded on a public ledger called the blockchain. Anyone can read every transaction ever made — who sent what, to which address, and when. The addresses have no names attached to them, but the payment history is completely open.

In this workshop you are the digital forensic investigator on Case CC6010-2025-047. Three evidence files have been extracted from a seized laptop and placed in your working directory. Your job is to work through them step by step, run a Python analysis script, and build up a complete picture of what happened.

⚠️ All evidence in this lab is fictitious

All names, addresses, transaction IDs, and amounts in the evidence files are entirely made up for this academic exercise. Do not search for them on any real Bitcoin website.

📁 CASE BRIEFING
// Case Reference: CC6010-2025-047 · Evidence: Extracted artefacts · Tools: Kali Linux + Python 3

A financial institution has reported suspicious activity involving a subject known as phantom_ax. The subject is suspected of paying for stolen payment card data using Bitcoin.

A warrant was executed at the home of Alex Mercer. A laptop was seized and a forensic image was taken. Three files have been extracted from the image and placed in your working directory. Your task is to trace the Bitcoin payments, confirm the wallet on the seized device, and produce a formal forensic report. Your findings form the basis of your submission for this workshop.

The investigation covers three Bitcoin transactions: the suspect purchased Bitcoin through an online exchange, used it to pay a known address, and the leftover change returned to a second address in their own wallet. The browser history on the seized laptop contains entries on the same dates as the payments.

💡 Three things to keep in mind as you work

Exchange → payment → change address. These three transactions form a chain. Each one is a piece of evidence. Your job is to trace the chain and connect it to the person.

The wallet file ties the device to the transactions. If the wallet on the seized laptop contains the same addresses that sent the payments, you have a direct link between the suspect and the activity.

Browser history confirms the timeline. A visit to blockchain.com on the same day as a payment — checking that transaction — places the suspect actively managing their Bitcoin from this device on that date.

🔑
What wallet.dat actually contains
The file holds private keys — and useful information even when encrypted

Bitcoin Core stores its wallet in a file called wallet.dat using a Berkeley DB database format. Inside it: a pool of private keys (typically 101 pre-generated), their corresponding Bitcoin addresses, and information about the last block the wallet synchronised with. If the user set a passphrase, the private keys are stored encrypted — they cannot be read without it.

Importantly, the addresses are stored in an unencrypted section of the key pool even when the keys themselves are encrypted. This means the wallet file still reveals which Bitcoin addresses this wallet owns, even if the private keys are locked. The last-used block height also tells you exactly when the wallet was last active — and you can cross-reference that with the transaction timestamps in EXH-001.

What you should find in wallet_manifest.txt
  • Database format: Berkeley DB 4.8 — genuine Bitcoin Core installation. Version string: /Satoshi:25.0.0/
  • Encryption: YES (AES-256-CBC). The private keys cannot be read without the wallet passphrase. The addresses, however, are stored in an unencrypted section and are directly readable.
  • Addresses recovered: 4 addresses in the unencrypted key pool. Address 1 = 1MercerAX9fGV2kQtPDyNz3QWaT7rBuKop · Address 2 = 1MercerCH4nGe7xYpLmV8rWqN2sT9bZKuw
  • Last sync: Block 812,389 · 2024-11-08 21:15:44 UTC — this matches the block height of Transaction #3 in EXH-001, confirming the wallet was active at the time of the final transaction.
  • Address linkage: Address 1 (1MercerAX...) is the input and a previous output in EXH-001. Address 2 (1MercerCH...) is the change output in Transaction #2. Finding both in the wallet file on the seized device proves the same device held the keys for both — the same person controlled the payment and the change.
TASK 1.2 — Read transactions.json (EXH-001) 10 marks
What to find in this file
  • A plain-English summary of what happened in each of the three transactions
  • The two outputs in Transaction #2 — which is the payment and which is the change?
  • The connection between the change address in TX #2 and the wallet file in EXH-003
🔄
What a change address is and why it matters
It belongs to the sender — and it links two addresses to the same wallet

In Bitcoin you cannot spend just part of an amount — you must use the full value of a previous output. If the suspect has 0.14985 BTC and wants to pay 0.09 BTC, the system takes the full 0.14985 BTC as input and creates two new outputs: 0.09 BTC to the recipient, and 0.05973 BTC back to the sender at a different address. That second address — the one receiving the leftover — is the change address.

The change address is forensically important because it belongs to the same wallet as the input. If that address also appears in the suspect's wallet file on the seized device, you have proven both addresses are controlled by the same person on the same machine.

// Simple analogy
You pay for a £9 coffee with a £20 note. You get £11 change back. That £11 is still yours — it came from your wallet and returns to your pocket. In Bitcoin, the leftover goes to your "change address". An investigator who finds that change address in your wallet file knows both addresses — the one you paid from and the change address — belong to you.
What you should find in transactions.json
  • TX #1 (2024-11-03): The exchange address sent 0.14985 BTC to the suspect's primary address (1MercerAX...). This is the suspect funding their wallet from an online exchange.
  • TX #2 (2024-11-05): The suspect's address sent 0.14985 BTC into the transaction. Two outputs were created: 0.09 BTC to the marketplace address (1DarkMkt...) and 0.05973 BTC back to 1MercerCH... as change. A miner fee of 0.00012 BTC was paid.
  • TX #3 (2024-11-08): The marketplace address consolidated its funds and sent 0.32385 BTC to address 1MixerPool... — consistent with a mixing service being used to obscure the trail.
  • Change address: In TX #2, output 1 (1DarkMkt..., 0.09 BTC) is the payment. Output 2 (1MercerCH..., 0.05973 BTC) is the change — it goes back to the sender. We know this because 1MercerCH... appears in the wallet key pool in EXH-003. A wallet only holds keys it generated itself, so this address belongs to the same person who sent the payment.

TASK 3.1 — Bitcoin Addresses and Private Keys 10 marks
📌 Reminder from this investigation

wallet_manifest.txt says the private keys are encrypted with AES-256-CBC. The addresses were recoverable from the unencrypted portion of the key pool. Both Address 1 and Address 2 appear in the transaction chain.

What a good answer covers
  • Key derivation (Q1): Private key (256-bit random number) → ECDSA → 64-byte public key → SHA-256 → RIPEMD-160 → 20-byte hash → Base58Check encoding → Bitcoin address (starts with 1 for P2PKH, or 3 for P2SH). Each step is a one-way function — you can go from private key to address, but you cannot reverse it.
  • Encrypted wallet (Q2): Options are: (1) use bitcoin2john to extract the hash, then Hashcat or John the Ripper with a dictionary attack (e.g. rockyou.txt) or brute-force; (2) search physical and digital exhibits for the passphrase written down, in a notes app, or in a password manager. Even without the keys, the investigation still has: the addresses from the unencrypted pool, the full on-chain transaction history, and the last sync timestamp — all of which link the device to the payments.
TASK 3.2 — Bitcoin Mixing — Transaction #3 5 marks
Transaction #3 from EXH-001 — the mixing step
JSON

What a good answer covers
  • What mixing does: A mixer takes Bitcoin from multiple senders, pools it, and sends different coins to output addresses. The one-to-one link between input and output is broken — you cannot directly trace which input corresponds to which output.
  • Why it is used: To make it harder for investigators to follow the money and connect a receiving address to the original sender.
  • What is still available: Chain analytics tools (Chainalysis, Elliptic) use pattern analysis and address clustering to sometimes identify mixer outputs. The exchange that funded Transaction #1 holds account records — this is the most productive remaining step. And the device evidence already gathered (wallet, browser history, transaction chain) is sufficient for the case against the suspect regardless of what happens to the funds after the mixer.
TASK 3.3 — Why Seizing the Hard Drive Is Not Enough 5 marks
What a good answer covers
  • No balance file exists: A Bitcoin balance is not stored anywhere — it is a calculation. To know how much an address holds, you scan the blockchain and sum all unspent transaction outputs (UTXOs) pointing to it. The hard drive holds the private key (in wallet.dat), which is what allows funds to be spent — not the coins themselves.
  • Without the passphrase: If the wallet passphrase is never recovered, the private keys remain encrypted and unreadable. The investigator can see the address and balance on the blockchain but cannot move or seize the funds. Knowing the balance is useless without the key.
  • What else to look for: Seed phrases (12 or 24 common English words written on paper, photographed, or in a notes app) which regenerate all keys from scratch. Hardware wallets (Ledger, Trezor devices). The passphrase in a password manager, email, or cloud notes. Other devices (phones, USB drives) that may hold a wallet copy.

⚖️
What makes a forensic findings section defensible
Every claim must have a source. Vague language is not acceptable.

A forensic report is not a story — it is a record of evidenced fact. Every claim must reference an exhibit number, a file path, a hash, or a specific output. "The wallet was found on the laptop" is too vague. "wallet.dat was recovered at C:\Users\amercer\AppData\Roaming\Bitcoin\wallet.dat, SHA-256: [value], consistent with EXH-003" is forensically correct — it is verifiable.

Use measured language. Write "the evidence shows" not "it is obvious that". Write "browser history (EXH-002) contains an entry at 14:10 on 2024-11-05" not "the suspect was definitely checking their wallet". The findings section states what you found — the conclusion subsection is where you connect it all together.

What a complete findings section includes
  • Section 1 — Artefacts: wallet_manifest.txt (EXH-003) from C:\Users\amercer\AppData\Roaming\Bitcoin\, SHA-256: 4f9a2b1d..., Berkeley DB 4.8, Bitcoin Core 25.0.0, encrypted. transactions.json (EXH-001) — three transactions with block heights, timestamps, and addresses. browser_history.csv (EXH-002) — 13 entries from Chrome and Tor Browser, 2024-10-28 to 2024-11-08.
  • Section 2 — Transaction trace: TX #1 (Block 812,044, 2024-11-03): exchange → suspect_primary 0.14985 BTC. TX #2 (Block 812,201, 2024-11-05): suspect_primary → marketplace 0.09 BTC + suspect_change 0.05973 BTC. Change address 1MercerCH... confirmed in wallet key pool (EXH-003). TX #3 (Block 812,389, 2024-11-08): marketplace → mixer 0.32385 BTC. Block height matches wallet last sync in EXH-003.
  • Section 3 — Browser correlation: 2024-11-05 14:10:03 Chrome visit to blockchain.com querying txid b2c3d4e5... — 27 minutes before TX #2 confirmed at 14:37 UTC. Two Tor Browser sessions recorded 2024-11-04 23:11 and 23:29 — the evening before the payment. 2024-11-08 21:00:01 Chrome visit to suspect_change address matching wallet last sync at 21:15:44 UTC.
  • Section 4 — Conclusion: wallet.dat (EXH-003) contains addresses suspect_primary and suspect_change, both appearing in EXH-001 transaction chain. Payment of 0.09 BTC on 2024-11-05 originated from suspect_primary, change returned to suspect_change — both in the same wallet on the seized device. Browser history places the device actively monitoring these transactions on the same dates. Passphrase recovery recommended.
