🧠 WHAT IS MEMORY FORENSICS? — READ FIRST

This section exists because many students jump straight to the commands and never understand what they are doing. If you skip this part, the investigation will feel like following a recipe without knowing what you are cooking. Take five minutes to read it.

What is RAM?

Your computer has two types of storage. Your hard drive (or SSD) is long-term storage — it holds everything even when the computer is switched off: your files, your installed programs, your photos. RAM (Random Access Memory) is completely different. It is your computer's short-term working memory — fast, temporary, and wiped completely when power is lost.

// Analogy — think of it this way
Imagine a chef working in a kitchen. The fridge and cupboards are the hard drive — food stored long-term. The countertop is RAM — only the things the chef is actively using right now are on it. The moment the chef finishes their shift, the countertop is cleared. RAM works exactly like that.

When you open a web browser, the browser loads from your hard drive into RAM to run. When you type a password into a website, that password exists in RAM while you are typing it. When you open a Word document, the entire document lives in RAM while it is open. When you run commands in a terminal window, those commands and their output live in RAM.

Turn the computer off — and all of that is gone forever.

What is a Memory Dump?

A memory dump (also called a RAM dump or memory image) is a snapshot photograph of everything in RAM taken at a specific instant in time. Special forensic tools can capture this snapshot while the computer is still running, freezing every piece of active data into a single large file.

ℹ️ In this lab

The file you will download — MemoryDump_Lab1.raw — is exactly this. It is a frozen snapshot of a Windows 7 computer's RAM captured at a specific moment in 2019. Everything that computer was doing at that moment is locked inside this file, waiting for us to read it.

Why Does RAM Matter as Evidence?

RAM contains things that the hard drive often does not. Consider these examples:

💬
Commands typed in a terminal
Even if the window was closed, the history stays in RAM until the computer restarts

When someone opens a command prompt (the black window) and types instructions, Windows stores those commands and their output in a memory buffer. If the window is closed, the window disappears — but the memory buffer does not. It stays in RAM. A forensic investigator can read it from a memory dump.

🔑
Passwords and encryption keys
Once a user logs in, Windows keeps their password hash in RAM permanently until shutdown

Windows does not store your actual password — it stores a scrambled version called a hash. But to let you access network resources and files without re-typing your password constantly, Windows caches this hash in RAM at all times. A forensic investigator can extract these hashes from a dump — and sometimes use them directly to access protected files.

📂
Open files and their contents
Files open in RAM even if the application was closed before saving

When you open a file — a Word document, an image, an archive — its contents are loaded into RAM. If the computer crashed before you saved, the file might still be recoverable from the dump. This is why memory forensics can recover evidence that was never written to the hard drive at all.

What is Volatility?

Volatility is a free, open-source tool that analyses memory dump files. It knows exactly how Windows organises data inside RAM — where the process list is, where network connections are recorded, where terminal histories are stored — and it can extract any of that information on demand.

// Analogy
A memory dump without Volatility is like having a massive library where every book has had its spine removed and all the pages mixed together. Volatility is the librarian who knows exactly where every section is and can retrieve any book in seconds.

Volatility uses plugins — each plugin is designed to extract one specific type of data. You will use four plugins in this lab:

imageinfo
Identifies which version of Windows this dump came from
pslist
Lists every program that was running at the time of capture
consoles
Reads the terminal window history — what was typed and what appeared on screen
hashdump
Extracts cached Windows password hashes from kernel memory
📁 THE CASE — READ BEFORE ANYTHING ELSE

Every real forensic investigation starts with a briefing. Read this carefully. The details in a briefing are clues — they tell you where to look.

// Case Reference: MemLab-01 · Memory Image Acquired: 2019-12-11

A computer belonging to a user named Alissa Simpson crashed unexpectedly. Before it crashed, a witness reported seeing a black terminal window briefly appear on screen. The witness could not read what was in it before it disappeared.

At the time of the crash, Alissa appeared to be using WinRAR — a program that opens compressed archive files — along with other applications.

A memory dump was captured from the machine immediately before it was turned off. Your task is to analyse this dump and recover any hidden evidence.

💡 Reading the clues in the briefing

Black terminal window → Someone ran a command in cmd.exe. Even if they closed the window, the command history lives in RAM. The consoles plugin will read it.

WinRAR was open → An archive file was being opened. Which file? Volatility can tell us. Is it password-protected? Volatility can help with that too.

🎯 WHAT ARE WE LOOKING FOR — AND WHY?

This is the question most labs never answer. Before you type a single command, you should know what you are looking for and why that thing matters.

Flag 1 — A Secret Hidden in the Terminal

The briefing tells us a terminal window appeared briefly. Terminals are used to run commands. Commands produce output. That output was displayed on screen — and it is still in RAM.

We are looking for something unusual in the terminal output. In this lab, whoever used the terminal printed a hidden message — encoded in Base64 to disguise it. Base64 is a way of writing text that makes it look like random characters. The string ZmxhZ3t0aDFz... is not random — it is a real message in disguise. Once we find it, we decode it with one command to reveal Flag 1.

🔍 What is Base64?

Base64 converts any text into a string of letters, numbers, and symbols. It is reversible — not encrypted — meaning anyone can decode it instantly. You will recognise Base64 strings because they often end with one or two = signs and look like meaningless characters. Example: ZmxhZw== decodes to flag.

Flag 2 — Inside a Password-Protected Archive

WinRAR was open with an archive file. That archive is password-protected — but we do not need to guess the password. Windows had already authenticated Alissa Simpson when she logged in, and it cached her password hash in RAM at that moment. We can extract her hash and use it to unlock the archive.

Inside the archive is a file containing Flag 2.

⚙️ SETUP — KALI LINUX 2025

Before You Start: Why Does Setup Take Several Steps?

Volatility 2 was written for Python 2.7 — a version of Python that reached end-of-life in 2020. Modern Kali Linux (2025) still has Python 2.7 installed, but it does not include the development files that are needed to install Python 2 extensions from source code. This means a clean Kali 2025 install will produce multiple errors when you try to set up Volatility.

The setup steps below fix these errors in the correct order. The order matters — doing them out of sequence will cause the errors to reappear.

💡 Opening a terminal in Kali

Right-click on the desktop and select Open Terminal, or find the terminal icon in the taskbar. The terminal is a text interface — you type a command and press Enter to run it. If you make a mistake, nothing permanent happens — you can just type again.

Step S1
Install Python 2.7 development headers
🤔
Why do we need this?

Python 2.7 is installed on Kali, but the C development header files — technical files needed to compile Python extensions from source code — are not. Without them, any library that needs to be compiled (like distorm3) will fail with a cryptic error about a missing file called Python.h. This step installs those missing files.

Kali Terminal — type this first
LINUX
sudo apt install python2.7-dev python2-dev -y

Type your Kali password when asked. You will not see the characters as you type — this is normal Linux behaviour.

Expected result
  • You should see Setting up python2.7-dev and Setting up python2-dev near the end
  • No error messages — just installation progress
Step S2
Install distorm3 — a library Volatility needs
🤔
Why do we need this?

distorm3 is a disassembly library — it helps Volatility analyse machine code found in memory. We install it now while the development headers from Step S1 are in place. Without Step S1 first, this would fail with the error shown below.

🚫 Error you would see WITHOUT Step S1

If you try this before installing dev headers, you will see:

Error output
ERROR
python/python_module_init.c:2:10: fatal error: Python.h: No such file or directory
compilation terminated.
error: command 'x86_64-linux-gnu-gcc' failed with exit status 1
ERROR: Failed building wheel for distorm3

This is exactly why Step S1 must come first.

Kali Terminal
LINUX
python2.7 -m pip install --user distorm3==3.4.4
Expected result — build succeeds this time
  • You will see a deprecation warning about Python 2.7 — ignore it, this is expected
  • You should see Building wheel for distorm3 ... done
  • Final line: Successfully installed distorm3-3.4.4
Step S3
Download Volatility 2 from GitHub
🤔
Why git clone and not apt install?

Kali's package manager (apt) does have a Volatility package, but it is an older version that may behave differently to the guide. Cloning from GitHub gives you the exact version used in this lab. On Linux, Volatility runs as a Python script — you call it with python2.7 vol.py rather than as a standalone program.

Kali Terminal — three commands in sequence
LINUX
cd ~
git clone https://github.com/volatilityfoundation/volatility.git
cd volatility

cd ~ goes to your home folder. git clone downloads Volatility. cd volatility enters that folder. Every command for the rest of this lab must be run from inside this folder.

Check it worked
  • Run ls and you should see a file called vol.py — that is the main Volatility program
  • Your terminal prompt should now show (kali㉿kali)-[~/volatility]
Step S4
Install pycryptodome — fixes a wall of error messages
🤔
Why do we need this?

Many Volatility plugins use cryptography functions — things like reading registry keys and decoding password hashes. These functions require a Python library called Crypto. Without it, Volatility still runs, but many plugins — including hashdump which we need for Flag 2 — will silently fail to load.

🚫 Error wall you would see WITHOUT this step

If you skip this, running any Volatility command shows:

Kali Terminal — from inside ~/volatility/

⚠ Common Mistakes at This Stage

🚫 Mistake 1 — Typing just "volatility" instead of "python2.7 vol.py"
What you typed (WRONG)
WRONG
┌──(kali㉿kali)-[~/volatility]
└─$ volatility -f MemoryDump_Lab1.raw imageinfo

Why: Volatility 2 is a Python script, not a system command. You must always prefix it with python2.7 vol.py. Typing just volatility only works if it was installed via apt — and we deliberately did not do that.

🚫 Mistake 2 — Running dumpfiles without creating the output folder first

Why: Volatility will not create the output folder for you. If it does not exist, the dumpfiles command fails immediately.

🔍 THE INVESTIGATION — STEP BY STEP

From this point on, every step has three parts: (1) what we are trying to find out and why, (2) what command to run, (3) what to look for in the output. Read all three before you type anything.

STEP 1 — IDENTIFY THE OS
What we are trying to find out
  • Before Volatility can read anything meaningful, it needs to know which version of Windows this dump came from
  • Different versions of Windows store data in different memory locations — Volatility uses an OS-specific map called a profile
  • Without the right profile, every other command will either fail or give wrong results
  • This is always the very first command in any memory forensics investigation — no exceptions
Run this
Ask Volatility to identify the OS profile
Kali — from inside ~/volatility/
LINUX
python2.7 vol.py -f ~/Desktop/MemoryDump_Lab1.raw imageinfo

This takes 30–60 seconds. Volatility is scanning the dump to find Windows kernel signatures. Wait until you see output appear.

Your output will look like this
OUTPUT
Volatility Foundation Volatility Framework 2.6.1
INFO    : volatility.debug    : Determining profile based on KDBG search...
          Suggested Profile(s) : Win7SP1x64, Win7SP0x64, Win2008R2SP0x64
                     AS Layer1 : WindowsAMD64PagedMemory (Kernel AS)
                     AS Layer2 : FileAddressSpace (.../MemoryDump_Lab1.raw)
                      PAE type : No PAE
                           DTB : 0x187000L

      Image local date and time : 2019-12-11 20:08:00 +0530
What to look for
  • The line Suggested Profile(s) — the first one listed is what we use: Win7SP1x64
  • Win7 = Windows 7 · SP1 = Service Pack 1 · x64 = 64-bit — this is the OS map Volatility will use
  • Write down Win7SP1x64 — you will add --profile=Win7SP1x64 to every command from now on
  • The image date tells us the dump was captured on 11 December 2019 at 14:38 UTC
STEP 2 — LIST RUNNING PROGRAMS
What we are trying to find out
  • Every running program on a computer is called a process. Windows keeps a list of all processes in memory
  • We want to see what was open when the computer crashed — this confirms the clues in our briefing
  • We are specifically looking for cmd.exe (the terminal window the witness saw) and WinRAR.exe (the archive program)
  • Each process has a unique number called a PID (Process ID) — like a name badge for each running program
Run this
List all processes that were running
Kali Terminal
LINUX
python2.7 vol.py -f ~/Desktop/MemoryDump_Lab1.raw --profile=Win7SP1x64 pslist
💡 Save long output to a file

The process list is long and will scroll past quickly. Add > ~/Desktop/pslist.txt at the end to save it to a file, then open it in a text editor:
python2.7 vol.py ... pslist > ~/Desktop/pslist.txt

Key lines from the output
OUTPUT
Offset(V)          Name              PID    PPID   Start
...
0xfffffa800...     explorer.exe      1064   2004   (the Windows desktop)
0xfffffa800...     cmd.exe           1984   1064   ← THE TERMINAL WINDOW
0xfffffa800...     WinRAR.exe        1512   1064   ← THE ARCHIVE PROGRAM
0xfffffa800...     mspaint.exe       2424   1064   (drawing program — not used today)

What to look for
  • cmd.exe — confirmed. The terminal the witness saw. Its PID is 1984
  • WinRAR.exe — confirmed. An archive was being opened. Its PID is 1512
  • Notice that both have the same PPID (Parent PID) — 1064 — which is explorer.exe, the Windows desktop. This means both were launched by the user from the desktop, not by the system
STEP 3 — READ THE TERMINAL HISTORY → FLAG 1
What we are trying to find out
  • Someone opened a terminal (cmd.exe) and typed something. We want to know exactly what they typed and what appeared on screen
  • Windows stores terminal input and screen output in a memory buffer called a console buffer
  • This buffer persists in RAM even after the terminal window is closed — it only disappears when the computer restarts
  • We expect to find something unusual in the output — a Base64-encoded flag hidden in the screen text
Run this
Read what was typed and displayed in the terminal
Kali Terminal
LINUX
python2.7 vol.py -f ~/Desktop/MemoryDump_Lab1.raw --profile=Win7SP1x64 consoles
Output — this is what you will see
OUTPUT
ConsoleProcess: conhost.exe Pid: 2104
Console: 0xff756200 CommandHistorySize: 50

  CommandHistory: Application: cmd.exe
    CommandCount: 1 LastAdded: 0
    Commands:
    cmd #0 at 0x...: St4G3$1

  Screen 0x... X:80 Y:300
    Dump:
    Microsoft Windows [Version 6.1.7601]
    ...
    ZmxhZ3t0aDFzXzFzX3RoM18xc3Rfc3Q0ZzMhIX0=
What to look for
  • Under Commands: the command typed was St4G3$1 — this is what they ran in the terminal
  • In the Dump section (the screen content): a long string ending in = — this is Base64-encoded text
  • The string: ZmxhZ3t0aDFzXzFzX3RoM18xc3Rfc3Q0ZzMhIX0=
  • It ends with = — the tell-tale sign of Base64. This is our flag in disguise

Decode the Base64 string to reveal Flag 1

Now we decode that string. Copy the exact string from your output — starting with Zmx and ending with the = sign — and run this:

Kali Terminal — decode the hidden message
LINUX
echo ZmxhZ3t0aDFzXzFzX3RoM18xc3Rfc3Q0ZzMhIX0= | base64 -d
🚫 Copy the string from YOUR output — not from here

The string above is from the example. Your output might look identical, but always copy from your own terminal. Do not add any spaces, line breaks, or extra characters. The string must start with a capital Z and end with the = sign. If you see base64: invalid input, you have copied it incorrectly.

⚑ Flag 1 — Found in Terminal Memory
flag{th1s_1s_th3_1st_st4g3!!}
What just happened? Someone ran a command called St4G3$1 in the terminal. That command printed a Base64-encoded string to the screen. Even though the terminal window was long gone, its entire screen buffer was still sitting in RAM. We found it with the consoles plugin and decoded it in one command. This is why clearing your screen — or closing the terminal — does not protect you. The memory buffer stays until the computer restarts.
STEP 4 — FIND THE HIDDEN ARCHIVE FILE
What we are trying to find out
  • We know WinRAR was running. But which file was it opening? A program can only be opened with a specific file — we need to find the filename and where it is stored
  • The cmdline plugin shows the full command used to launch each process, including any files that were passed to it as arguments
  • Once we know the filename, we use filescan to find its memory address — then extract it with dumpfiles
Run this first
Find what file WinRAR was opening
Kali Terminal
LINUX
python2.7 vol.py -f ~/Desktop/MemoryDump_Lab1.raw --profile=Win7SP1x64 cmdline
Find the WinRAR entry
OUTPUT
...
*************************************
WinRAR.exe pid:   1512
Command line : "C:\Program Files\WinRAR\WinRAR.exe"
             "C:\Users\Alissa Simpson\Documents\Important.rar"
...
What to look for
  • Find the WinRAR.exe section
  • The file it was told to open: Important.rar in Alissa Simpson's Documents
  • This is the archive we need to extract from memory and open

Find the file's memory address

Now we know the filename. We use filescan to scan all file handles in memory and filter for this specific file. The result gives us the memory address — the exact location of this file's data in RAM.

Kali Terminal — scan and filter
LINUX
python2.7 vol.py -f ~/Desktop/MemoryDump_Lab1.raw --profile=Win7SP1x64 filescan | grep -i "important"
💡 What does | grep do?

filescan produces thousands of lines — one for every file that was open. The | grep -i "important" part filters it: it only shows lines containing the word "important". The | symbol (called a pipe) passes the output of one command into another. The -i flag makes the search case-insensitive.

Output — one matching line
OUTPUT
0x000000003fa3ebc0    16    0  R--r--  \Device\HarddiskVolume2\Users\Alissa Simpson\Documents\Important.rar
What to look for
  • The long number at the start: 0x000000003fa3ebc0 — this is the memory address of the file
  • The 0x prefix means hexadecimal (base-16 numbering) — just copy it exactly as it appears
  • Write this down — you need it in the next command

Extract the file from memory

Now we pull the actual file data out of the memory dump using that address and save it to our output folder.

Run this
Extract Important.rar from the dump
Kali Terminal
LINUX
python2.7 vol.py -f ~/Desktop/MemoryDump_Lab1.raw --profile=Win7SP1x64 dumpfiles -Q 0x000000003fa3ebc0 -D ~/Desktop/output/
⚠️ Use the address from YOUR output

The address 0x000000003fa3ebc0 is from this guide's example run. The address in your output should match — but if it differs, use yours.

Expected output
OUTPUT
Volatility Foundation Volatility Framework 2.6.1
DataSectionObject 0x3fa3ebc0   None   \Device\...\Important.rar

Go to ~/Desktop/output/ in your file manager. You will see a file with a long name ending in .dat. Rename it to Important.rar — the .dat extension is just how Volatility names extracted files, but the content is the original archive.

STEP 5 — GET THE PASSWORD AND OPEN THE ARCHIVE → FLAG 2
What we are trying to find out
  • Important.rar is password-protected. We need the password to open it
  • We are not going to guess — we are going to extract the password hash from memory
  • When Alissa logged into Windows, her password was hashed (scrambled) and stored in RAM by the kernel security process (LSASS). It stays there until the computer shuts down
  • The hashdump plugin reads those cached hashes directly from the LSASS process memory
  • The archive password is Alissa's NTLM hash — in uppercase
🔑
What is an NTLM hash — and why can we use it as a password?

Windows never stores your actual password. Instead it stores a hash — a scrambled, fixed-length version produced by a mathematical function. When you type your password to log in, Windows hashes what you typed and compares it to the stored hash. If they match, you are in.

For this lab, the person who created the archive set its password to be Alissa's NTLM hash (in uppercase). This was deliberate as part of the challenge. In real-world forensics, extracted hashes can be used in pass-the-hash attacks — authenticating as a user without knowing their real password, just by using the hash directly.

Run this
Extract Windows password hashes from memory
Kali Terminal
LINUX
python2.7 vol.py -f ~/Desktop/MemoryDump_Lab1.raw --profile=Win7SP1x64 hashdump
Output
OUTPUT
Volatility Foundation Volatility Framework 2.6.1
Administrator:500:aad3b435b51404eeaad3b435b51404ee:31d6cfe0d16ae931b73c59d7e0c089c0:::
Guest:501:aad3b435b51404eeaad3b435b51404ee:31d6cfe0d16ae931b73c59d7e0c089c0:::
Alissa Simpson:1000:aad3b435b51404eeaad3b435b51404ee:f4ff64c8baac57d22f22edc681055ba6:::
What to look for
  • Find the line for Alissa Simpson
  • The format is: Username : RID : LM Hash : NTLM Hash :::
  • The NTLM hash is the last long string before the :::
  • For Alissa Simpson: f4ff64c8baac57d22f22edc681055ba6
  • The archive password is this hash in UPPERCASE: F4FF64C8BAAC57D22F22EDC681055BA6

Open the archive using the hash as the password

The password to use — must be UPPERCASE
F4FF64C8BAAC57D22F22EDC681055BA6

Navigate to ~/Desktop/output/ and open Important.rar. When asked for a password, enter the hash above in uppercase. You can also use the terminal:

When prompted for a password, type: F4FF64C8BAAC57D22F22EDC681055BA6

⚑ Flag 2 — Found via Password Hash + Archive
flag{w3ll_3rd_stage_was_easy}
What just happened? Windows caches the NTLM hashes of logged-in users in a kernel process called LSASS (Local Security Authority Subsystem Service). This cache exists so Windows does not need to re-authenticate you constantly. We extracted Alissa's hash from that cache using hashdump and used it as the archive password — demonstrating that memory forensics can defeat password protection if the system was running and authenticated at the time of capture.
📋 WHAT DID WE ACTUALLY LEARN?

Stepping back — here is what this investigation demonstrated about digital forensics:

1 — RAM holds evidence the hard drive does not

Commands typed and then deleted, files opened but not saved, passwords entered but not stored — all of these exist in RAM. A memory dump can recover all of it.

2 — Closing a window does not erase memory

The terminal window was gone — but its console buffer (with the typed command and its output) stayed in RAM. Forensic tools read RAM directly, not the interface.
